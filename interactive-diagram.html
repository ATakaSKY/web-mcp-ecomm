Data Flow: Agent → Tool → UI

1

Agent discovers tools

Chrome exposes all registered tools via navigator.modelContext. The agent sees tool names, descriptions, and JSON input schemas.

↓
2

useWebMCP.ts registers 7 imperative tools

On mount, registerTool() is called for each tool. useRef keeps cart/wishlist state fresh inside closures without re-registering.

↓
3

Declarative forms auto-register

Forms with toolname attribute in DeclarativeView.tsx are always registered. QuickBuyModal's form registers only when the modal is open.

↓
4

Agent calls a tool

For imperative tools, the execute() callback runs directly. For declarative tools, Chrome auto-fills form fields and fires toolactivated.

↓
5

StoreContext dispatches state change

Tool execute functions call dispatch() which updates cart, wishlist, or UI state via the reducer. React re-renders the affected views.

↓
6

UI updates & result returns

Components re-render with new state. The tool returns structured content { content: [{ type: "text", text }] } (imperative) or respondWith(Promise) (declarative) back to the agent.

Special Flow: Two-Tool Pattern (Modal)

A

Agent calls open_quick_buy (imperative)

Dispatches OPEN_QUICK_BUY with a product ID. The reducer sets quickBuyProductId, which conditionally mounts the QuickBuyModal component.

↓
B

complete_quick_buy auto-registers (declarative)

The modal's <form toolname="complete_quick_buy"> enters the DOM. Chrome detects it and auto-registers the tool. The agent can now see and call it.

↓
C

Agent fills form → user confirms → respondWith(Promise)

No toolautosubmit, so the user must click "Confirm Purchase". On submit, e.agentInvoked detects the source and respondWith(Promise.resolve(...)) returns order data. The UI transition is delayed to avoid unmounting the form during processing.
