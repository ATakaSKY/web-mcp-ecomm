WebMCP Developer Guide

Expose structured tools on your website so AI agents interact with your UI reliably — no more screen-scraping.

What is it?

A proposed web standard that lets websites publish structured tools (JS functions or annotated HTML forms) for in-browser AI agents to discover and call.

Why does it matter?

Replaces brittle screen-scraping with a contract — agents know exactly what tools exist, what inputs they need, and what outputs to expect.

How do I use it?

Either write JavaScript (Imperative API) or add HTML attributes to your forms (Declarative API). Chrome exposes them via navigator.modelContext.

⬡ How WebMCP Works — Architecture Flow

↻ Tool Lifecycle

1

Register

Website registers tools via JS or HTML form annotations

2

Discover

Agent queries navigator.modelContext for available tools & schemas

3

Invoke

Agent calls tool with structured args; browser fires toolactivated

4

Execute

JS function runs or form auto-fills; CSS pseudo-classes activate

5

Return

Structured result sent back to agent; UI updates for the user

⇌ Two APIs — Side by Side

Imperative API

Full control via JavaScript 
// Register
navigator.modelContext.registerTool({
  name: "searchFlights",
  description: "Search flights",
  inputSchema: {
    type: "object",
    properties: {
      origin: { type: "string" },
      destination: { type: "string" }
    }
  },
  execute: ({ origin, destination }) => {
    // your logic here
    return { content: [{ type: "text",
      text: `Results for ${origin}→${destination}`
    }]};
  }
});

// Unregister
navigator.modelContext.unregisterTool("searchFlights");
  • Full programmatic control over tool behavior
  • Custom execute function — you handle the logic
  • Dynamically register / unregister based on app state
  • Returns structured results directly

Declarative API

HTML attributes on existing forms 
<!-- Just add attributes to your form -->
<form
  toolname="search_flights"
  tooldescription="Search available flights"
  toolautosubmit
  action="/search">

  <label for="from">Origin airport</label>
  <input name="from" type="text" />

  <select name="class"
    toolparamdescription="Cabin class">
    <option value="economy">Economy</option>
    <option value="business">Business</option>
  </select>

  <button type="submit">Search</button>
</form>
  • Zero JS required — just annotate existing HTML
  • Browser auto-generates JSON schema from form fields
  • Agent auto-fills fields; user clicks submit (unless toolautosubmit)
  • Use toolparamdescription to override label text

✦ Declarative API — Events, CSS & Extras

FeatureName / SelectorDescription
Event toolactivated Fires on window once the agent fills form fields. Non-cancelable. Has toolName property.
Event toolcancel Fires when the user cancels the agent operation or form.reset() is called.
SubmitEvent e.agentInvoked Boolean — true when a form submit was triggered by an AI agent (not a human).
SubmitEvent e.respondWith(promise) Pass a promise that resolves to tool output. Must call e.preventDefault() first.
CSS Pseudo :tool-form-active Applied to the <form> element while the agent is interacting with it.
CSS Pseudo :tool-submit-active Applied to the form's submit button during agent interaction.
Attribute toolautosubmit If set, form auto-submits after agent fills fields (no user click needed).
Attribute toolparamdescription Overrides the label-derived description for a form field in the generated schema.

★ Tool Declaration Best Practices

1

Naming & Semantics

  • Use specific verbs: create-event vs start-event-creation-process
  • Describe what the tool does, not what it doesn't do
  • Positive descriptions > negative limitations
2

Schema Design

  • Accept raw user input — don't make the agent do math
  • Use explicit types: string, number, enum
  • Explain why behind enum options, not just the values
3

Error Recovery

  • Validate strictly in code, loosely in schema
  • Return descriptive errors so the model can self-correct
  • Handle rate limits gracefully with clear messages
  • Return after UI has updated (agents check UI state)
4

Tool Philosophy

  • Keep tools atomic & composable — one function each
  • Don't overlap — combine similar tools via params
  • Trust the agent's flow control — avoid rigid sequencing rules
  • Register/unregister dynamically based on page state

⬡ MCP vs WebMCP

MCP (Model Context Protocol)

Server-side. Developers deploy tools on their own server. AI agents connect via a server-side protocol. Requires backend infrastructure.

vs

WebMCP

Client-side. Tools live in the browser as JS functions or annotated HTML forms. AI agents interact directly with the page — no separate server needed.

⚠ Current Limitations

Browser Tab Required

Tools need a visible browsing context — no headless/background execution supported.

UI Sync Is Your Job

You must keep UI state updated after tool calls, just as you would for human interactions.

Complexity Overhead

Complex UIs may need refactoring to properly handle state and outputs for agent interactions.

No Discovery Mechanism

Agents can't discover which sites offer tools without visiting them. No registry or search integration yet.

Note: This is an early preview behind a flag in Chrome 146. Expect bugs — Google is actively seeking developer feedback to shape the spec.
